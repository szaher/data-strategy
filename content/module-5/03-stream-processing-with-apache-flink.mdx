## Overview

In the last lesson, you learned how **Apache Spark** handles batch processing with DataFrames and partitioned transformations. Batch processing works brilliantly when your data arrives in discrete chunks — daily exports, hourly logs, weekly snapshots. But what happens when events stream in continuously, and you need answers in seconds, not hours?

This is the domain of **stream processing**, where systems like **Apache Flink** shine. Stream processing treats data as an unbounded flow of events rather than fixed datasets. The challenge isn't just speed — it's handling **out-of-order arrival**, **late data**, and **stateful computations** while guaranteeing correctness and fault tolerance.

Here's the problem: if you use the time when your system *processes* an event (processing time) instead of when the event *actually happened* (event time), your results become non-deterministic. Reprocess the same stream twice, and you'll get different answers. This is unacceptable for production analytics, fraud detection, or billing systems.

Flink solves this by making **event time** a first-class citizen. It uses **watermarks** to track progress through event time, **windows** to group events into bounded chunks, and **checkpointed state** to enable exactly-once processing guarantees. This lesson will show you how these primitives work together to build correct, fault-tolerant streaming pipelines.

You'll learn the difference between event time and processing time, how watermarks signal event-time progress, when to use tumbling vs sliding vs session windows, how Flink's state and checkpointing enable exactly-once semantics, and how to handle late-arriving data without corrupting results.

By the end, you'll know how to build streaming applications that produce deterministic results, recover from failures without data loss, and handle real-world messiness like network delays and clock skew.

## Flink Fundamentals: Everything Is a Stream

Apache Flink's core philosophy is that **all data is a stream** — whether it's unbounded (live clickstream, sensor telemetry) or bounded (historical logs, CSV files). This unified mental model means you write the same DataStream API code for both cases, and Flink handles the execution details.

**Unbounded streams** flow continuously with no defined end. Examples: user clicks, IoT sensor readings, database change events (CDC). These streams require special handling for time, state, and windowing.

**Bounded streams** have a start and an end. Examples: a Parquet file, yesterday's logs, a batch of transactions. Flink treats these as streams that happen to terminate. You can apply the same windowing and stateful operations, but Flink can apply batch-specific optimizations like sorting before joins.

The **DataStream API** is Flink's core abstraction. You define a **source** (Kafka, Kinesis, files, sockets), apply **transformations** (map, filter, keyBy, window, aggregate), and write to a **sink** (databases, files, downstream queues). Flink compiles this into a directed acyclic graph (DAG) of operators and executes it across a cluster.

Here's the minimal structure of a Flink streaming job:

```
Source → Transformation(s) → Sink
```

Unlike Spark, which executes in micro-batches, Flink is a **true streaming engine**. Events flow through the operator pipeline one at a time (or in small batches for efficiency). This gives Flink sub-second latency for many workloads.

**Flink on Kubernetes** has become the de facto deployment model. The **Flink Kubernetes Operator** automates lifecycle management: deploying JobManager and TaskManager pods, managing savepoints, handling upgrades. You choose between two modes:

- **Session mode**: Multiple jobs share a single Flink cluster. Good for resource efficiency but jobs interfere with each other.
- **Application mode**: Each job gets its own dedicated cluster. Isolation is perfect, but overhead is higher.

For production streaming workloads, **application mode** is the safe default. Session mode is fine for dev/test or when you have dozens of lightweight jobs.

Flink's architecture separates the **JobManager** (coordinator) from **TaskManagers** (workers). JobManager handles scheduling, checkpointing, and failure recovery. TaskManagers execute the actual data processing. This separation allows Flink to scale horizontally by adding more TaskManagers.

## Event Time vs Processing Time: The Correctness Problem

Imagine you're tracking click events from a mobile app. A user clicks at 10:00:00 AM, but their phone has poor connectivity. The event sits in a local queue for five minutes, then uploads at 10:05:00 AM. Your streaming system receives it at 10:05:02 AM.

Which timestamp matters? The click happened at **10:00:00** (event time), but your system processed it at **10:05:02** (processing time).

If you count "clicks per minute" using **processing time**, this click is counted in the 10:05 bucket. Reprocess the same stream tomorrow, and maybe network conditions are different — now the event arrives at 10:04:58 and lands in the 10:04 bucket. Same input, different output. **Non-deterministic**.

If you count using **event time**, the click always lands in the 10:00 bucket, regardless of when it arrived. Reprocess the stream a thousand times, and you get the same answer. **Deterministic**.

Event time is the recommended default for most analytics because it produces deterministic results, but it introduces a new problem: **how do you know when you've seen all events for a given time period?** If you're computing the count for 10:00–10:01, how long do you wait before finalizing the result? Events can arrive out of order or late.

This is where **watermarks** come in.

Think of a watermark as a signal that says, "I've seen all events up to time T." It's like a teacher starting class: they wait for a few minutes after the bell rings to give latecomers a chance, then close the door. The door closing is the watermark — it marks the boundary between "on time" and "late."

Flink generates watermarks based on the timestamps embedded in your events. A common strategy is **bounded-out-of-orderness watermarks**: "I assume events arrive no more than N seconds late." For example, if the latest event timestamp is 10:05:00 and your allowed lateness is 10 seconds, Flink emits a watermark for 10:04:50. Based on the configured lateness assumption, this watermark tells downstream operators that events with timestamps at or before 10:04:50 are unlikely to arrive. Later arrivals must follow the configured late-data policy (drop, side output, or allowed lateness).

If an event with timestamp 10:04:45 arrives after the watermark for 10:04:50 has passed, it's **late data**. Flink can either drop it, process it separately (side outputs), or allow a grace period (**allowed lateness**) before finalizing the window.

**Processing time** still has valid use cases: when you care about *when the system observed the event* rather than when it happened. Example: monitoring pipeline lag, alerting on "no events received in the last 5 minutes," or approximations where determinism isn't critical. But for business logic — aggregations, analytics, machine learning features — always use event time.

## Watermarks: Tracking Event-Time Progress

Let's make watermarks concrete with an analogy. Imagine you're running a relay race where runners hand off batons at checkpoints. Some runners are fast, some are slow. You're the judge at the finish line, and you need to know when all batons for "heat 1" have crossed.

Each baton has a label: "heat 1," "heat 2," etc. (this is the event timestamp). Runners arrive out of order — heat 1 batons might arrive mixed with heat 2. You wait until you're confident all heat 1 batons have arrived before declaring heat 1 complete. That confidence signal is the watermark.

In Flink, a **watermark** is a special record that flows through the streaming pipeline. It carries a timestamp T and signals the pipeline's estimate that events with timestamp ≤ T are unlikely to arrive — a heuristic based on configured lateness assumptions, not an absolute guarantee. Watermarks are generated at sources and propagate downstream through operators.

Flink supports multiple watermark strategies:

**Bounded-out-of-orderness:** Assumes events arrive within a fixed delay. If you set max delay to 10 seconds, a watermark for time T is emitted when the source sees an event with timestamp T + 10 seconds. This is the most common strategy.

**Monotonous watermarks:** For perfectly ordered streams (rare in practice), watermarks can advance immediately with each event.

**Custom strategies:** You can write arbitrary watermark logic if you have domain-specific knowledge (e.g., events from certain sources are always delayed by exactly 5 minutes).

Here's the key insight: watermarks are **heuristic**. They're a best-effort signal based on assumptions about your data. If you set max lateness to 10 seconds but an event arrives 15 seconds late, Flink will treat it as late data. You can configure **allowed lateness** to keep windows open for a grace period, but eventually you must finalize results.

**Parallelism and watermarks:** When Flink runs with multiple parallel instances of an operator, each instance generates its own watermarks. Downstream operators take the **minimum** watermark across all upstream partitions. This ensures correctness: a window only fires when *all* upstream sources have advanced past the window's end time.

This minimum-watermark behavior has an important consequence: **one slow partition stalls the entire pipeline**. If one Kafka partition is lagging or one data source is delayed, watermarks won't advance, and windows won't fire. This is a common production issue — monitor watermark lag across partitions and investigate stragglers.

**Idle sources:** If a partition stops sending events (e.g., low-traffic time of day), its watermark stops advancing, blocking downstream progress. Flink provides an **idle source timeout** — if no events arrive within N seconds, the partition is marked idle and excluded from minimum-watermark computation.

## Windows: Grouping Events for Aggregation

Windows are the fundamental building block for stateful aggregations over unbounded streams. Without windows, how would you compute "average response time" over an infinite stream? Windows slice the stream into finite chunks that you can aggregate.

Flink supports four window types:

### Tumbling Windows

Fixed-size, non-overlapping windows. Each event belongs to exactly one window.

Example: "Count events per 1-minute window." Windows are [00:00–00:01), [00:01–00:02), [00:02–00:03), etc.

Use tumbling windows for periodic reporting: hourly metrics, daily summaries, fixed-interval aggregations.

### Sliding Windows

Fixed-size, overlapping windows. Each event can belong to multiple windows.

Example: "Compute average response time over the last 5 minutes, updated every 1 minute." Windows are [00:00–00:05), [00:01–00:06), [00:02–00:07), etc. A request at 00:02 appears in five windows.

Use sliding windows for moving averages, rolling metrics, or when you need smooth, continuously updated results rather than discrete buckets.

### Session Windows

Variable-size windows based on activity gaps. A session starts with the first event and ends after a period of inactivity (the "gap").

Example: "Group user clicks into sessions, where a session ends after 10 minutes of inactivity." If a user clicks at 00:00, 00:02, 00:03, then goes silent until 00:15, you get one session [00:00–00:13) (last click at 00:03 + 10-minute gap).

Use session windows for user behavior analysis, fraud detection (transaction bursts), or any domain where natural activity breaks define boundaries.

### Global Windows

All events belong to a single window. You must provide a custom trigger to decide when to fire.

Use global windows only for specialized cases where none of the above fit. Most pipelines use tumbling, sliding, or session.

**Window assignment** happens before aggregation. Flink looks at each event's timestamp (event time) or processing time and assigns it to one or more windows. Then Flink applies your aggregation function (sum, count, custom reduce) to each window.

**Window triggers** determine when a window fires. For event-time windows, the default trigger fires when a watermark passes the window's end time. You can customize triggers for early results (emit partial counts before the window closes) or late-firing (emit updated results when late data arrives).

## State and Checkpointing: Exactly-Once Guarantees

Stateful stream processing is what makes Flink powerful. **State** is any information an operator needs to remember across events. Examples:

- Counters per key (e.g., "number of logins for user_id=123")
- Sliding-window aggregations (keep the last N events)
- Partial aggregates for open windows
- Machine learning model parameters being updated incrementally

Flink provides two types of state:

**Keyed state:** Scoped to a specific key. After calling `keyBy(...)`, each parallel instance of an operator maintains state only for the keys it's responsible for. Flink automatically partitions and scales keyed state.

**Operator state:** Scoped to a parallel operator instance, not tied to individual keys. Less common; used for source offsets or custom partitioning logic.

Keyed state is what you'll use 90% of the time. Flink provides convenient state primitives:

- **ValueState\<T\>**: Stores a single value per key.
- **ListState\<T\>**: Stores a list of values per key.
- **MapState\<K, V\>**: Stores a map per key.
- **ReducingState\<T\>** and **AggregatingState\<T\>**: Optimized for incremental aggregations.

State lives in memory (heap or off-heap) for fast access. For large state, Flink uses **RocksDB** as the state backend — a local embedded database that spills to disk.

**State TTL (Time-To-Live)** is critical in production. If your state grows unbounded, you'll run out of memory. TTL automatically expires state that hasn't been accessed or updated within a configured duration. Example: "Expire user session state after 1 hour of inactivity."

Without TTL, a keyed state operator will accumulate state for every unique key it ever sees. If you're tracking user activity and you have millions of users, this becomes a memory leak.

**Checkpointing** is how Flink achieves fault tolerance and exactly-once semantics. Periodically (e.g., every 60 seconds), Flink takes a **distributed snapshot** of all operator state and event positions in sources (Kafka offsets, Kinesis shard positions, file byte offsets).

The checkpoint process is asynchronous and uses a technique called **Chandy-Lamport snapshots**. Flink injects special **barrier** markers into the stream. When an operator receives a barrier, it snapshots its state and forwards the barrier downstream. This ensures all operators snapshot a consistent global state without pausing event processing.

Checkpoints are written to durable storage: S3, HDFS, or network-attached storage. If a TaskManager crashes, Flink restores the latest checkpoint and reprocesses events from the checkpointed source offsets. Because Flink replays from the exact same offsets and uses the same state snapshot, internal state is recovered consistently — **exactly-once semantics for Flink's internal state**. End-to-end exactly-once guarantees additionally depend on source replay capabilities, sink transactional support, and the absence of external side effects.

**Exactly-once vs at-least-once:** Flink's checkpoint mode defaults to exactly-once for internal state, but you can configure at-least-once (CHECKPOINT_MODE = AT_LEAST_ONCE) for lower overhead.

<VerifyClaim status="verify">

At-least-once is faster (less checkpointing I/O) but means duplicate processing on failures. For analytics and reporting, exactly-once is mandatory. For approximate monitoring, at-least-once may suffice.

**Recovery time** depends on checkpoint interval and state size. Smaller intervals mean faster recovery but higher overhead. Larger intervals mean lower overhead but longer recovery. Typical production setting: 1–5 minute intervals, 30–60 second timeout.

## Late-Arriving Data: Handling Reality's Messiness

Even with watermarks, data arrives late. Network partitions, clock skew, upstream delays, mobile devices offline for hours — the real world is messy. Flink provides three strategies for handling late data:

### 1. Drop Late Data (Default)

Once a watermark passes a window's end time and the window fires, Flink closes the window. Late events are discarded. This is safe but wasteful — you lose data.

### 2. Allowed Lateness

Keep the window state around for a grace period after firing. If late events arrive, recompute the aggregation and emit an **updated result**. Consumers must handle updates (idempotent writes, versioned records, etc.).

Example: Fire a 1-minute tumbling window when the watermark passes, but allow 30 seconds of additional lateness. A late event arriving 10 seconds after the watermark still updates the window.

Allowed lateness is configured per window:

```
.window(TumblingEventTimeWindows.of(Time.minutes(1)))
.allowedLateness(Time.seconds(30))
```

The tradeoff: you hold state longer (memory cost) and emit multiple results for the same window (downstream complexity).

### 3. Side Outputs

Route late data to a separate output stream for special handling: logging, dead-letter queues, manual review, or reprocessing with relaxed lateness bounds.

```python
late_output_tag = OutputTag("late-data", Types.STRING())

windowed_stream = (
    stream
    .window(TumblingEventTimeWindows.of(Time.minutes(1)))
    .side_output_late_data(late_output_tag)
    .apply(my_aggregation_function)
)

late_events = windowed_stream.get_side_output(late_output_tag)
```

Side outputs give you full visibility into late data without polluting the main results. You can monitor late-data volume, investigate causes, and adjust watermark strategies.

**Best practice:** Start with allowed lateness for business-critical aggregations (billing, reporting) and side outputs for monitoring. Drop late data only when you're confident your watermark strategy covers 99%+ of events.

## Flink Windowed Aggregation with Watermarks (Code Example)

Here's a complete PyFlink example showing event-time windows, watermarks, and keyed state:

<CodeBlock code={`from pyflink.datastream import StreamExecutionEnvironment, TimeCharacteristic
from pyflink.datastream.window import TumblingEventTimeWindows
from pyflink.common import Time, WatermarkStrategy, Types
from pyflink.common.serialization import SimpleStringSchema
from pyflink.datastream.connectors.kafka import KafkaSource
from pyflink.common.watermark_strategy import TimestampAssigner

# Define a custom timestamp extractor
class EventTimestampAssigner(TimestampAssigner):
    def extract_timestamp(self, value, record_timestamp):
        # Assume value is JSON: {"user_id": "123", "timestamp": 1623456789000, "event": "click"}
        import json
        event = json.loads(value)
        return event["timestamp"]  # milliseconds since epoch

env = StreamExecutionEnvironment.get_execution_environment()
env.enable_checkpointing(60000)  # Checkpoint every 60 seconds

# Define Kafka source with watermark strategy
kafka_source = (
    KafkaSource.builder()
    .set_bootstrap_servers("localhost:9092")
    .set_topics("user-events")
    .set_value_only_deserializer(SimpleStringSchema())
    .build()
)

watermark_strategy = (
    WatermarkStrategy
    .for_bounded_out_of_orderness(Time.seconds(10))  # Allow 10 sec lateness
    .with_timestamp_assigner(EventTimestampAssigner())
)

stream = env.from_source(kafka_source, watermark_strategy, "Kafka Source")

# Parse JSON and key by user_id
def parse_event(value):
    import json
    event = json.loads(value)
    return (event["user_id"], event["event"])

keyed_stream = stream.map(parse_event, output_type=Types.TUPLE([Types.STRING(), Types.STRING()])) \\
                     .key_by(lambda x: x[0])

# Count events per user in 1-minute tumbling windows
windowed_counts = (
    keyed_stream
    .window(TumblingEventTimeWindows.of(Time.minutes(1)))
    .allow_lateness(Time.seconds(30))  # Allow 30 sec late arrivals
    .apply(lambda key, window, events: (key, window.end, len(list(events))))
)

windowed_counts.print()

env.execute("Flink Event-Time Windowed Aggregation")
`} language="python" filename="flink_windowed_aggregation.py" />

This example:

- Reads from Kafka with a custom timestamp extractor that pulls event time from JSON payloads.
- Uses bounded-out-of-orderness watermarks with 10 seconds max lateness.
- Keys events by `user_id` and applies 1-minute tumbling windows.
- Allows 30 seconds of additional lateness — windows recompute if late events arrive.
- Emits `(user_id, window_end_timestamp, event_count)` tuples.

In production, you'd write results to a sink (database, Kafka, S3) instead of printing. You'd also configure state TTL if user_id cardinality is unbounded, and you'd set up monitoring for watermark lag and late-data volume.

## AI Cross-Cut: Real-Time Feature Engineering

Machine learning models in production often need **real-time features** — aggregations computed from the last N events or the last M minutes of activity. Flink is the natural choice for this workload.

### Use Case: Fraud Detection

You're building a fraud-detection model that predicts whether a credit card transaction is fraudulent. One critical feature is **"number of transactions in the last 10 minutes for this card."** This is a sliding-window count keyed by `card_id`.

In Flink:

```python
card_stream = env.from_source(kafka_source, watermark_strategy, "Transactions")

features = (
    card_stream
    .key_by(lambda txn: txn["card_id"])
    .window(SlidingEventTimeWindows.of(Time.minutes(10), Time.minutes(1)))
    .apply(lambda key, window, txns: {
        "card_id": key,
        "txn_count_10min": len(list(txns)),
        "total_amount_10min": sum(txn["amount"] for txn in txns),
        "window_end": window.end
    })
)
```

Every minute, Flink emits updated features for each card. These features are written to a low-latency store (Redis, DynamoDB) and read by the model serving layer when a new transaction arrives.

### Use Case: Personalization

You're personalizing a news feed based on recent user behavior. You need features like "articles read in the last hour," "categories viewed," "time spent per category."

Session windows are perfect for this:

```python
user_stream = env.from_source(kafka_source, watermark_strategy, "User Events")

sessions = (
    user_stream
    .key_by(lambda event: event["user_id"])
    .window(EventTimeSessionWindows.with_gap(Time.minutes(30)))
    .apply(lambda key, window, events: {
        "user_id": key,
        "session_duration": window.end - window.start,
        "articles_read": [e["article_id"] for e in events if e["type"] == "read"],
        "categories": list(set(e["category"] for e in events))
    })
)
```

Each session is a feature vector. Write these to your feature store, and the recommendation model pulls them when the user requests their feed.

### Checkpointed State for Incremental Models

Some models can be updated incrementally — online gradient descent, count-min sketches, HyperLogLog for cardinality estimation. Flink's keyed state is perfect for storing model parameters per user or per item.

Example: incrementally update a per-user embedding every time the user interacts with an item. The embedding is stored in `ValueState`, updated on each event, checkpointed for fault tolerance, and periodically written to a model store.

## Decision Framework: When to Use Which Window Type

| **Use Case**                              | **Window Type** | **Why**                                                                 |
|-------------------------------------------|-----------------|-------------------------------------------------------------------------|
| Hourly/daily metrics for reporting        | Tumbling        | Fixed intervals, no overlap, easy to reason about                       |
| Moving average over last N minutes        | Sliding         | Smooth, continuously updated aggregations                               |
| User session analysis                     | Session         | Natural activity boundaries, variable length                            |
| Real-time dashboards (latest 5-min avg)   | Sliding         | Update frequently, show trend over recent window                        |
| Billing/invoicing (exact time buckets)    | Tumbling        | Deterministic, non-overlapping periods                                  |
| Fraud detection (burst detection)         | Session         | Detect rapid-fire activity within short gaps                            |
| Model feature computation (last N events) | Sliding or Count| Sliding for time-based, count-based for event-based                     |

**State TTL decision:**

- **Short TTL (minutes to hours):** User sessions, temporary caches, short-lived features.
- **Long TTL (days):** User profiles, long-term aggregations, model parameters.
- **No TTL:** Only when cardinality is strictly bounded (e.g., keyed by a fixed set of sensors).

## Failure Modes and How to Avoid Them

### 1. Using Processing Time for Business Logic

**Symptom:** Reprocessing the same input produces different results. Aggregations shift when you replay after a failure.

**Fix:** Always use event time for analytics, billing, reporting. Reserve processing time for operational metrics (pipeline lag, alerting).

### 2. Unbounded State Growth

**Symptom:** TaskManagers run out of memory over time. State size grows linearly with unique keys.

**Fix:** Configure state TTL. Aggressively expire state that's no longer needed. Monitor state size per operator.

### 3. Ignoring Late Data

**Symptom:** Undercounting or missing events. Monitoring shows a small but steady stream of late arrivals.

**Fix:** Use allowed lateness or side outputs. Monitor late-data volume. Adjust watermark max-lateness based on observed 99th-percentile delays.

### 4. Watermark Lag from Slow Partitions

**Symptom:** Windows don't fire, even though most partitions are caught up. One slow Kafka partition blocks the entire pipeline.

**Fix:** Enable idle source timeout. Monitor per-partition watermark lag. Investigate slow partitions — they may have network issues, skewed data, or upstream delays.

### 5. Checkpoint Timeout

**Symptom:** Checkpoints fail frequently, Flink keeps retrying, pipeline falls behind.

**Fix:** Increase checkpoint timeout, reduce checkpoint frequency, or switch to incremental checkpoints (RocksDB). Profile state size and I/O throughput.

### 6. Non-Deterministic Watermark Generation

**Symptom:** Reprocessing produces slightly different window boundaries or counts.

**Fix:** Ensure timestamp assignment is deterministic — extract timestamps from event payloads, not system clocks. Avoid relying on processing-time-based watermark adjustments.

## Summary

Apache Flink enables **correct, fault-tolerant stream processing** by treating event time as a first-class citizen. Watermarks signal progress through event time, allowing windows to fire deterministically. Keyed state and checkpointing enable exactly-once semantics with fast recovery from failures.

The core primitives — **event time**, **watermarks**, **windows**, **stateful operators**, and **checkpoints** — compose into arbitrarily complex streaming pipelines. Tumbling windows handle periodic reporting, sliding windows smooth trends, and session windows capture natural activity patterns.

Late data is inevitable in production. Handle it with **allowed lateness** (recompute and update results) or **side outputs** (route to special handling). Never ignore late data without measuring its impact.

For AI workloads, Flink is the engine of choice for **real-time feature computation**. Sliding-window aggregations feed fraud-detection models, session windows power personalization, and checkpointed state enables incremental model updates.

The failure modes to watch: using processing time for business logic, unbounded state growth, ignoring late data, watermark lag from stragglers, and checkpoint timeouts. Monitor watermark progress, state size, late-data volume, and checkpoint success rates.

Flink's learning curve is steeper than Spark's, but the reward is **deterministic, low-latency, exactly-once stream processing** — the foundation of modern real-time analytics and AI systems.
